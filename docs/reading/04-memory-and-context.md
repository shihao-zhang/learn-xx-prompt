# 04 Memory 与 Context：连续性也需要边界

## 学习目标

区分上下文和记忆，理解为什么 Agent 需要连续性，也为什么连续性必须可控、可验证、可删除。

## 四类上下文

读 Agent prompt 时，可以把上下文分成四类：

- 当前会话：用户刚刚说的话、工具刚返回的结果。
- 运行环境：日期、工作目录、权限模式、可用工具。
- 项目上下文：仓库文件、issue、PR、测试日志、设计文档。
- 长期记忆：用户偏好、项目约束、过去确认过的工作方式。

它们的权威性不同。通常系统指令高于用户消息，用户最新消息高于旧消息，真实文件高于记忆摘要。

## 读法

一些 prompt 会规定记忆文件格式、索引、类型、写入时机和删除规则。PM 应重点关注：

- 记什么：用户偏好、项目长期约束、外部资源链接。
- 不记什么：代码本身已有的信息、一次性对话、敏感数据。
- 何时更新：用户明确要求、旧记忆被证明过期、项目目标变化。
- 如何验证：引用记忆前要检查事实是否仍然成立。
- 如何删除：记忆错误或用户撤回时应可清理。

记忆不是“越多越聪明”。错误记忆会让 Agent 自信地做错事。

## Context compaction

长会话会被摘要或压缩。PM 需要把它当成产品能力来设计，而不是技术细节。

要问：

- 压缩后 Agent 是否知道自己在继续同一个任务？
- 摘要会不会丢失最新用户指令？
- 恢复后是否会重新验证关键文件和状态？
- 用户是否能理解 Agent 为什么继续某个步骤？

如果 prompt 写着“恢复后继续自然工作”，那评估时就要测试长任务恢复。

## 关键问题

- 记忆的写入是否需要用户明确意图？
- 记忆和项目事实冲突时，Agent 听谁的？
- 哪些上下文来自 harness 注入，哪些来自用户？
- 长期记忆是否有类型、来源和更新时间？
- 是否能避免把工具结果里的恶意文本当作高优先级指令？

## 练习

为 GitHub 项目 Agent 写一条记忆策略草案：

1. 可以保存 3 类信息。
2. 禁止保存 3 类信息。
3. 记忆过期时如何更新。
4. 用户要求“忘记”时如何处理。
5. 使用记忆前如何验证。

示例方向：

- 可以保存：用户偏好的 PR 描述格式、项目发布流程、固定测试命令。
- 不应保存：密钥、临时失败日志、仓库里已经明确记录的普通代码结构。
- 使用前验证：如果记忆提到文件、命令、分支或服务，先检查它是否仍存在。

## PM 小结

上下文让 Agent 会工作，记忆让 Agent 像在持续协作。但产品上必须保留边界：来源清楚、优先级清楚、过期可改、错误可删。
