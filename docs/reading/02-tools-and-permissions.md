# 02 工具与权限：能力不是越多越好

## 学习目标

学会把工具列表读成产品能力地图，并理解权限系统如何把 Agent 从“会说”变成“可控地做事”。

## 工具怎么读

每个工具都可以用六个问题理解：

- 这个工具解决什么用户任务？
- 输入参数是什么？
- 输出结果是什么？
- 会不会改变本地文件、远程服务或用户状态？
- 失败时 Agent 能否恢复？
- 是否需要用户确认或更高权限？

常见工具类别：

- 读取类：读文件、搜代码、看网页、查任务。
- 写入类：编辑文件、生成文档、更新计划、创建 PR。
- 执行类：运行 shell、测试命令、启动服务。
- 外部类：浏览器、GitHub、网络搜索、定时任务。
- 协作类：子 Agent、后台任务、workflow。

PM 要重点看“副作用”。读文件通常低风险，删除文件、发布内容、推送代码、访问外部服务通常高风险。

## 权限怎么读

权限不是阻碍 Agent，而是产品信任的一部分。好的权限设计会回答：

- 默认能做什么？
- 哪些操作必须确认？
- 用户拒绝后 Agent 怎么继续？
- 是否能记住某类授权？
- 是否避免把一次授权扩展到无关场景？

例如，代码 Agent 常见规则包括：

- 不随意执行破坏性命令。
- 不回退别人改动。
- 网络受限时需要明确授权。
- 对外发布、删除、覆盖等动作要更谨慎。
- 权限失败后调整方案，而不是原样反复请求。

## 读法

建议做一张表：

| 工具 | 用户价值 | 副作用 | 权限要求 | 失败恢复 | 需要埋点 |
| --- | --- | --- | --- | --- | --- |
| 读文件 | 理解项目 | 低 | 默认允许 | 换路径/搜索 | 读取失败率 |
| 编辑文件 | 完成修改 | 中 | 工作区内允许 | 回滚补丁/重试 | 修改文件数 |
| shell | 测试/构建 | 中到高 | 受 sandbox 限制 | 换命令/请求授权 | 命令耗时、失败码 |
| GitHub | PR/issue 操作 | 高 | 账号和权限 | 要求重新认证 | API 失败率 |

这张表比单纯罗列工具更有产品价值。

## 关键问题

- 工具名称是否让用户和 Agent 都能理解它的用途？
- schema 是否限制了危险输入？
- 权限拒绝后的体验是否清楚？
- Agent 会不会用低层 shell 绕过更安全的专用工具？
- 高风险操作有没有确认、预览或可撤销机制？

## 练习

为 GitHub 项目里的“自动修复文档错别字”设计权限策略：

1. 列出允许默认执行的工具。
2. 列出必须确认的工具。
3. 写一条用户拒绝权限后的 Agent 回复原则。
4. 写一条防止误删或覆盖他人修改的规则。
5. 写一个验收标准：怎样证明这套权限策略有效？

参考答案方向：

- 读文件、搜索、编辑指定目录可默认允许。
- 推送分支、创建 PR、删除文件、访问外网需要确认。
- 用户拒绝后应说明影响，并提供本地可完成的替代方案。

## PM 小结

工具定义能力上限，权限定义信任下限。Agent 产品的成熟度，常常体现在“能做很多事，但不会乱做事”。
