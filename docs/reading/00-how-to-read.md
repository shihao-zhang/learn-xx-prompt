# 00 如何阅读 Agent Prompt

面向对象：AI 产品经理、增长/平台 PM、负责 Agent 体验和能力边界的人。

源材料入口：
- Phistory 站点：https://phistory.cc/
- Phistory GitHub：https://github.com/WEIFENG2333/phistory
- 本项目数据源索引：`data/phistory-sources.json`

## 学习目标

读完后，你应该能把一份 Agent system prompt 看成产品规格，而不是“提示词技巧”。它同时规定了：

- Agent 承诺完成什么用户目标。
- Harness 提供哪些输入、工具、权限和上下文。
- Agent 在不确定、失败、危险操作、长任务时如何行动。
- 产品如何评估“做完了”和“做得可靠”。

## 读法

先不要逐句精读。第一遍只标出五类句子：

- `must`：不可违反的硬约束。
- `should/prefer`：默认策略。
- `never/do not`：风险边界。
- `tool schema`：Agent 能调用的真实能力。
- `handoff/reporting`：什么时候问用户、什么时候汇报结果。

第二遍把这些句子翻译成 PM 语言：

- 这是用户价值，还是内部操作规范？
- 这是模型行为，还是 harness 能力？
- 失败时用户看到什么？
- 这个规则能不能被测试？

第三遍看版本差异。Phistory 的价值不只是保存原文，而是让你观察 agent 产品怎样演化：更强的自主性、更细的权限、更明确的记忆规则、更可验证的完成标准。

## 关键问题

- 这个 Agent 的核心工作是什么？写代码、浏览网页、长期记忆、还是多 Agent 协作？
- 哪些行为由模型自己判断，哪些行为由工具和权限系统强制约束？
- prompt 是否把“何时停下、何时确认、何时继续”讲清楚？
- 用户看到的是自然语言回复、文件改动、GitHub PR、浏览器状态，还是多种结果的组合？
- 这个 prompt 的质量能否通过测试用例判断，而不只是凭感觉？

## 练习

从 `data/phistory-sources.json` 里选一份 prompt 源链接，例如 Codex CLI 或 Claude Code，完成一页标注：

1. 找出 5 条硬约束。
2. 找出 5 个工具或工具类别。
3. 找出 3 个需要用户确认的场景。
4. 找出 3 个“完成任务前必须验证”的要求。
5. 用一句话写出这个 Agent 的产品定位。

输出格式建议：

```markdown
产品定位：
硬约束：
工具能力：
确认场景：
验证要求：
我认为最值得测试的一条规则：
```

## 读完标志

你能把一份 prompt 拆成三张图：用户体验图、工具权限图、评估用例图。做到这一点，才算真正读懂。
