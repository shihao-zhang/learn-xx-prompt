# 05 Evaluation：把 Prompt 变成可测试的产品行为

## 学习目标

学会为 Agent prompt 设计评估，不只看回答质量，也看工具使用、权限边界、任务完成度、恢复能力和用户信任。

## 评估对象

Agent prompt 的评估至少包含五层：

- 任务成功：是否真正解决用户问题。
- 工具选择：是否用了合适工具，是否避免不必要风险。
- 权限安全：是否在高风险操作前确认，拒绝后是否调整。
- 工作流质量：是否探索、计划、执行、验证、汇报。
- 上下文连续性：是否正确处理新消息、历史摘要和长期记忆。

这五层合起来，才是 Agent 产品质量。

## 读法

读 evaluation 相关内容时，把每条规则改写成“输入、期望行为、失败信号”。如果一句 prompt 不能转成测试，它可能只是风格偏好；如果能转成测试，它就是产品契约。

## 测试用例模板

```markdown
用例名称：
用户请求：
初始环境：
允许工具：
禁止或需确认的操作：
期望行为：
通过标准：
失败信号：
需要观察的日志/埋点：
```

PM 写评估时，不要只写“回答应该正确”。要写出可观察行为，例如：

- 是否读取了相关文件。
- 是否只改指定范围。
- 是否运行了测试。
- 是否说明测试结果。
- 是否没有推送远程分支。
- 是否在权限拒绝后给出替代方案。

## GitHub 项目场景

可以从这些场景开始评估：

| 场景 | 重点观察 |
| --- | --- |
| 修复一个小 bug | 是否最小修改、运行相关测试、解释结果 |
| 代码审查 | 是否先列风险和行号，而不是泛泛总结 |
| 文档更新 | 是否只改文档范围、保持术语一致 |
| CI 失败 | 是否读取日志、定位失败、避免盲改 |
| 权限被拒绝 | 是否尊重拒绝并换成本地方案 |
| 上下文恢复 | 是否继续最新任务，而不是旧任务 |
| 记忆冲突 | 是否验证事实，而不是相信旧记忆 |

## 版本差异怎么评

Phistory 适合做版本对比。你可以选同一个 Agent 的两个版本，观察：

- 新版本增加了哪些工具？
- 是否新增权限或 sandbox 规则？
- 自主性是增强还是收敛？
- 是否更强调验证和测试？
- 是否改变了最终回复风格？
- 是否新增记忆、子 Agent、workflow 或长期任务能力？

把这些差异转成回归测试。只要 prompt 改了，产品行为就可能改。

## 关键问题

- 这个 prompt 最容易造成哪类误操作？
- 哪些规则必须自动化测试，哪些适合人工评审？
- 测试是否覆盖“工具失败”和“用户拒绝”？
- 是否评估了成本、耗时和用户打断？
- 是否能解释为什么某个版本更适合当前产品定位？

## 练习

设计 10 条评估用例，覆盖：

- 2 条只读问答。
- 2 条本地文件修改。
- 2 条需要运行测试。
- 2 条高风险权限确认。
- 1 条上下文压缩后继续任务。
- 1 条长期记忆与当前文件冲突。

每条都用上面的模板写。最后给每条用例打标签：

```text
success / safety / autonomy / memory / reporting / cost
```

## PM 小结

Prompt 不是交付物的终点，而是评估的起点。真正成熟的 Agent 产品，会把 prompt 里的每条重要规则变成测试、监控或用户可见的交互设计。
