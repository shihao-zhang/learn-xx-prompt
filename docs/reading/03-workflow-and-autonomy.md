# 03 Workflow 与 Autonomy：自主性要有停靠点

## 学习目标

理解 Agent 的自主性不是“永远自动执行”，而是一套可预测的工作流：什么时候行动、什么时候询问、什么时候验证、什么时候停下。

## 工作流的基本形状

在多个 Agent prompt 中，你会反复看到类似循环：

```text
理解任务 -> 探索上下文 -> 制定计划 -> 执行修改 -> 运行验证 -> 汇报结果
```

这就是代码 Agent 的产品骨架。区别只在于：

- 有的 Agent 更强调先计划。
- 有的 Agent 更强调持续执行直到完成。
- 有的 Agent 更强调多 Agent 并行。
- 有的 Agent 更强调每一步向用户同步。

PM 要看的不是谁“更积极”，而是谁的停靠点更清楚。

## 自主性分级

可以把 Agent 行为分成四级：

- L0 直接回答：不读文件、不改状态。
- L1 只读探索：搜索、读取、总结，不改文件。
- L2 本地修改：在明确范围内编辑、测试、生成文件。
- L3 外部行动：推送、发 PR、发消息、创建定时任务、访问外部账号。

越往上，越需要明确授权、可见反馈和恢复策略。

## 读法

读 workflow/autonomy 段落时，标出触发条件：

- 用户只是提问时，是否直接回答？
- 用户要求修改时，是否默认动手？
- 任务很大时，是否先计划？
- 需要危险操作时，是否确认？
- 测试失败时，是否继续修复？
- 用户发来新消息时，是否以最新消息为准？
- 上下文被压缩或恢复后，是否继续原任务？

这些触发条件就是产品交互规则。

## 关键问题

- Agent 的默认动作会不会超出用户预期？
- 什么情况下它应该停下来问问题？
- 什么情况下它应该不问，直接完成？
- “完成”是否包括验证，还是只包括代码写完？
- 多 Agent 或后台任务会不会让用户失去控制感？

## 练习

把下面 4 个请求分别归入 L0-L3，并写出 Agent 应该怎么做：

1. “解释这个函数为什么慢。”
2. “把 README 的安装步骤改清楚。”
3. “修复 CI，并开一个 PR。”
4. “删除旧分支和所有临时文件。”

建议输出：

```markdown
请求：
自主性等级：
是否需要确认：
推荐工具：
验证方式：
用户可见汇报：
```

然后补充一条产品规则：

> 当任务包含外部发布、删除、账密、付费资源或不可逆操作时，Agent 必须先确认。

## PM 小结

自主性不是一个开关，而是一组状态机。优秀的 Agent 会主动完成低风险工作，同时在高风险边界前停得很稳。
