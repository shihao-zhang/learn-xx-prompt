# 01 Harness：Agent 产品的运行外壳

## 学习目标

理解 harness 是什么，以及为什么 AI 产品经理不能只看模型能力。Agent 的实际表现来自“模型 + prompt + harness + 工具 + 权限 + UI”的组合。

## 概念解释

Harness 可以理解为 Agent 的运行外壳。它负责把用户消息、系统指令、环境信息、工具结果、权限状态和最终输出串起来。

在 Phistory 的多份 prompt 里，你会看到类似元素：

- 用户输出会以 Markdown、终端文本、浏览器或文件形式呈现。
- 工具调用有结构化 schema，不是模型随口说说。
- 环境上下文会被注入，例如工作目录、日期、权限模式。
- 某些标签或提醒来自系统，不应被当作普通用户输入。
- 文件引用、行号、任务状态、后台进程都可能由 harness 管理。

对 PM 来说，harness 决定了“产品能做什么”，也决定了“出了问题怎么被用户理解”。

## 读法

读 harness 相关段落时，画一条链路：

```text
用户请求 -> 系统/环境注入 -> Agent 判断 -> 工具调用 -> 权限检查 -> 结果回传 -> 用户可见输出
```

每读到一个规则，就问它落在哪个环节。例如：

- “优先使用专用文件工具”属于 Agent 判断。
- “权限被拒绝后不要原样重试”属于权限检查和失败恢复。
- “工具结果可能包含系统提醒”属于结果回传的信任边界。
- “引用文件要带行号”属于用户可见输出。

## 关键问题

- 哪些上下文是用户主动给的，哪些是系统注入的？
- Agent 能否区分用户指令、系统提醒、工具结果和历史摘要？
- 工具失败时，产品希望 Agent 重试、换方案、还是交还给用户？
- 最终输出是否足够说明“我做了什么、验证了什么、还有什么风险”？
- harness 有没有给 PM 留下可观察指标，例如工具成功率、权限拒绝率、恢复次数？

## 练习

假设用户在 GitHub 项目中说：“修复这个失败的 CI。”

请用 8 步写出 harness 流程：

1. 用户请求进入会话。
2. harness 注入仓库、权限、日期、工具列表。
3. Agent 读取失败信息。
4. Agent 决定需要哪些工具。
5. 权限系统允许或拒绝工具调用。
6. Agent 根据工具结果修改文件。
7. Agent 运行验证。
8. Agent 汇报结果和剩余风险。

然后补充两个失败分支：

- 权限被拒绝时，用户应该看到什么？
- 测试仍失败时，Agent 应该如何报告？

## PM 小结

Prompt 是行为契约，harness 是执行契约。一个 Agent 产品如果只优化 prompt，而不设计 harness 的输入、权限、反馈和失败恢复，用户体验会很难稳定。
