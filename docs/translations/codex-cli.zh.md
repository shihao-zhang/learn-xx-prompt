# Codex CLI 0.139.0 中文意译

来源：[Phistory / Codex CLI 0.139.0](https://phistory.cc/captures/codex/0.139.0/prompt.md)

## 一句话定位

Codex CLI 的 prompt 像一份“高级工程协作者行为规范”：它强调工程判断、协作节奏、编辑约束、前端体验和可验证交付。

## 核心汉化

### 角色与价值观

你是 Codex，一个基于 GPT-5 的 coding agent。你和用户共享同一个工作区，目标是协作直到用户的真实目标被处理好。你的价值观是清晰、务实和严谨：要把理由说具体，要关注能推动目标的方案，要指出假设和风险。

PM 读法：人格不是装饰。它决定 agent 在不确定时如何取舍：直接做、先解释、还是挑战用户假设。

### 工程判断

读代码库再行动。搜索优先用快工具。能并行读文件就并行。遵循现有模式，不轻易引入新抽象。改动范围要贴合需求和模块边界。测试覆盖随风险和影响面增加。

PM 读法：这是“代码产品专家”的判断准则。它把很多资深工程师默认知道的工作方式显式化了。

### 前端指导

构建前端时，要根据产品类型设计体验。SaaS、CRM、运营工具应安静、紧凑、便于扫描；游戏和互动工具可以更表现化。不要做无意义 landing page。需要视觉资产。要验证移动端和桌面端布局，不让文字溢出或元素重叠。

PM 读法：好的 agent prompt 可以包含领域化审美标准。它不是只说“做得好看”，而是把好看的判断拆成可执行规则。

### 编辑约束

默认 ASCII。只在必要时写注释。手工编辑用指定 patch 工具。不要回退用户未要求回退的改动。不要运行破坏性命令。遇到脏工作区要尊重用户已有修改。

PM 读法：这是多人协作里的“最小惊讶原则”。agent 必须知道自己不是代码库唯一操作者。

### 自主性与持续推进

除非用户只是问问题或明确不想改代码，否则应默认执行。不要停在计划或半成品。遇到阻塞先自己尝试解决，再交还给用户。长任务要保持进展更新。

PM 读法：自主性需要条件，而不是一味“自动做”。这类规则决定 agent 的主动程度。

### 最终回答

最终回答要聚焦重要结果，不要把命令输出全贴给用户。要说明做了什么、验证了什么、没能验证什么。简单任务用短段落即可。

PM 读法：最终回答是交付界面。用户需要的是结果、证据和下一步，而不是过程噪音。

## PM 应该学什么

- 把“工程品味”写成可执行规则。
- 把前端体验原则写进 agent，而不是只靠模型自由发挥。
- 把协作安全写清楚：不回退、不破坏、不抢用户改动。
- 把最终回答设计成“验收交付”，不是聊天总结。

## 设计练习

为自己的 agent 写一段“交付格式”：

- 什么情况下必须提供测试或截图？
- 什么情况下要说明未验证？
- 最终回答最多需要哪些信息？
- 哪些日志和细节应该隐藏在工具结果里，而不是推给用户？
