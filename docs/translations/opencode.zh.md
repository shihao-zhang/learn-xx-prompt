# opencode 1.17.6 中文意译

来源：[Phistory / opencode 1.17.6](https://phistory.cc/captures/opencode/1.17.6/prompt.md)

## 一句话定位

opencode 的 prompt 是强执行型 agent 的典型样本：它反复强调“继续推进直到问题解决”，并把调研、计划、实现、测试写成线性流程。

## 核心汉化

### 高自主承诺

你是 opencode，一个 agent。用户问题没有完全解决前，不要结束回合。你应该自主解决问题，不要轻易把控制权交回给用户。说要调用工具，就真的调用工具；说要做某事，就继续做。

PM 读法：这是强自主体验的核心承诺。优点是推进感强，风险是容易过度操作，所以必须配合权限和验证机制。

### 必须调研

如果用户提供 URL，要先抓取。涉及第三方包、依赖、框架时，要联网确认最新用法。不能只靠过时知识。

PM 读法：这把“最新性”从模型能力问题转成产品流程问题：当事实会变时，agent 必须查证。

### 工作流

工作流程包含：抓 URL、深入理解、调查代码库、互联网研究、制定计划、增量修改、调试、频繁测试、迭代到根因解决、最后反思验证。

PM 读法：这是一条完整任务状态机。可以把它拆成产品中的步骤、进度条、检查点和验收门槛。

### 沟通

每次工具调用前，用一句话告诉用户接下来要做什么。复杂任务要用 todo list，并随着进展打勾。回答要清晰、直接、友好、专业。

PM 读法：进度可见性会影响用户是否信任 agent。过度沉默和过度输出都是体验问题。

### 测试

测试不足是失败主因。每次改动后要验证，考虑边界情况和隐藏测试。不要只修表象，要找根因。

PM 读法：agent 的“完成”不能只靠自己声称，而要有测试、复现和边界检查。

## PM 应该学什么

- 强自主 agent 需要明确停止条件。
- “必须查证”适合事实变化快的产品领域。
- todo list 是任务可视化，不只是内部管理。
- 测试和反思可以作为任务完成前的硬门槛。

## 设计练习

为你的产品定义“自主推进等级”：

- L1：只回答，不操作。
- L2：提出计划，等用户确认。
- L3：低风险自动执行，高风险确认。
- L4：完整自主执行，但最终必须给证据。

你的产品默认应该是哪一级？为什么？
