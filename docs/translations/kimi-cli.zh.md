# Kimi CLI 1.47.0 中文意译

来源：[Phistory / Kimi CLI 1.47.0](https://phistory.cc/captures/kimi/1.47.0/prompt.md)

## 一句话定位

Kimi CLI 的 prompt 重点是工具使用、编码规范、研究数据处理、计划模式和任务管理。它像一个结构清晰的 CLI agent 操作说明。

## 核心汉化

### Prompt 与工具使用

agent 需要理解不同消息层级和工具约束，按可用工具完成任务。遇到明确适用的技能或计划模式时，要进入对应流程。

PM 读法：工具不是简单的按钮，而是 prompt 行为的一部分。PM 要定义“什么时候用什么工具”。

### 编码通用指南

修改代码前先理解上下文。遵循已有风格。尽量小步修改。不要无关重构。必要时添加测试或验证。

PM 读法：这是 coding agent 的基础质量门槛，适合做成评审 checklist。

### 研究与数据处理

涉及外部事实、数据处理或研究任务时，要注意信息源、数据准确性和处理方法，不要凭空编造。

PM 读法：研究类 agent 的风险不是“不会写”，而是“看起来像真的”。所以来源和方法必须进入 prompt。

### 计划模式

Kimi CLI 有 EnterPlanMode 和 ExitPlanMode。计划模式用于多步骤、需要用户确认或方案权衡的任务。退出计划模式前要确保已经给出清晰方案。

PM 读法：计划模式是产品状态，不只是 agent 心里想一想。它改变了用户和 agent 的交互协议。

### Todo 与任务列表

复杂任务要用任务列表跟踪进展。任务项需要状态，便于用户理解当前工作。

PM 读法：todo 是用户可见的任务管理界面，可以降低长任务焦虑。

## PM 应该学什么

- 把“计划中”和“执行中”做成清晰状态。
- 把工具触发条件写具体。
- 把研究任务的事实查证写进 prompt。
- 让任务列表服务用户理解，而不是只服务 agent 内部思考。

## 设计练习

定义你产品里的 Plan Mode：

- 什么任务必须先计划？
- 计划里必须包含哪些字段？
- 用户确认后，哪些内容变成执行约束？
- 什么情况下需要退出计划、重新规划？
