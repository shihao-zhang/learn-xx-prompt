# SOTA Agent Prompt 结构分类

这份分类不是按原文顺序拆，而是按 AI 产品经理需要理解的产品模块拆。

## 1. 身份与任务边界

回答的问题：这个 agent 是谁，承诺解决什么问题，不做什么？

常见内容：

- 角色：coding agent、personal assistant、CLI assistant
- 工作对象：代码库、浏览器、文件、GitHub、远程节点
- 明确拒绝或限制：破坏性安全请求、越权操作、绕过审批、长期自我目标

产品含义：这是 agent 的定位页和风险边界。它决定用户会期待什么，也决定安全审核如何判断越界。

## 2. Harness 与运行环境

回答的问题：模型生活在什么产品容器里？

常见内容：

- 输出会被渲染成 Markdown、终端文本或应用内消息
- 工具调用可能被权限系统拒绝
- 系统注入的 reminder、trace、context 与用户消息不同
- 文件路径、行号、图片、媒体、线程状态有特定格式

产品含义：harness 是“模型和产品之间的接口说明”。PM 设计 agent 时，不能只写 prompt，还要设计权限、上下文、工具结果、渲染和恢复机制。

## 3. 工具与权限协议

回答的问题：agent 可以调用什么工具？什么时候调用？失败后怎么办？

常见内容：

- 文件读写、命令执行、浏览器、GitHub、搜索、图片生成、子代理
- 优先使用专用工具，而不是让用户手动操作
- 高风险命令需审批
- 工具失败后换查询、换路径、换来源，而不是原样重试

产品含义：工具协议是 agent 产品的“操作系统”。工具越强，越需要权限、回滚、审计和用户可见解释。

## 4. 工作流与自主性

回答的问题：agent 是只回答，还是会持续推进？

常见内容：

- 先理解问题，再调查，再计划，再实现，再验证
- 简单任务直接做，复杂任务拆计划
- 不在能继续推进时只给计划
- 长任务给进度更新
- 使用 todo、task、workflow 或 subagent 管理复杂工作

产品含义：自主性不是越高越好，而是“在正确边界内持续推进”。PM 要定义何时自动、何时停下、何时找用户确认。

## 5. 记忆与上下文管理

回答的问题：agent 记住什么？怎么防止记错或泄漏？

常见内容：

- 用户偏好、项目约束、历史反馈、长期记忆
- 新会话启动时加载索引或摘要
- 记忆要验证是否仍然有效
- 不保存代码库本身已经记录的信息

产品含义：记忆不是“全都记住”，而是“可索引、可更新、可删除、可验证”。PM 要把记忆当成产品功能设计。

## 6. 沟通与输出格式

回答的问题：用户如何感知 agent 的工作？

常见内容：

- 中间进展要短
- 最终回答要给结论、改动、验证
- 引用文件路径和行号
- 不要输出过度冗长的命令日志
- 不要用可见文案解释 UI 细节

产品含义：沟通规范影响信任。用户真正需要的是“我知道它在做什么，并且能验证结果”。

## 7. 验证与交付证据

回答的问题：怎样证明任务完成？

常见内容：

- 运行测试、lint、build
- 截图或浏览器验证
- 明确说明没有运行的检查
- 对失败输出如实汇报

产品含义：验证是 agent 的交付协议。没有证据的“完成了”，对 PM 来说只是一个 UI 文案，不是产品能力。

## 8. 安全、审批与外部影响

回答的问题：哪些动作会造成不可逆、外部发布或权限风险？

常见内容：

- 删除、覆盖、推送、发布、外部服务发送前要确认
- 不绕过审批
- 不扩大权限
- 不把一次授权泛化成永久授权

产品含义：安全 prompt 应该和产品权限系统一起设计。光靠一句“谨慎操作”不够，需要清晰触发条件和恢复路径。

## PM 评审清单

检查自家 agent prompt 时，可以逐项问：

- 是否定义了任务边界和拒绝边界？
- 是否说明了运行环境和输出格式？
- 是否列清楚工具选择策略？
- 是否定义了权限失败后的行为？
- 是否有“持续推进”和“停下询问”的判断标准？
- 是否有记忆的写入、读取、更新和删除策略？
- 是否要求最终交付带证据？
- 是否能从 prompt 规则映射到产品需求和指标？
