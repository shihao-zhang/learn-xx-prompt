# 模块覆盖矩阵

这张表帮助 AI 产品经理快速比较 7 个 Phistory 样本在关键 prompt 模块上的侧重。

评分不是排行榜，而是阅读导航：

- `强`：该模块是样本的突出特征，值得优先阅读。
- `中`：有明确规则，但不是最核心亮点。
- `弱`：规则较少或只作为基础能力出现。

## 怎么用

1. 先选一个你关心的模块，例如“权限安全”。
2. 读该列里标为 `强` 的样本。
3. 到 `docs/translations/` 查看中文意译。
4. 把规则转写成自己的产品需求。
5. 用 `docs/evaluation/agent-behavior-rubric.md` 做行为评估。

## 关键观察

- Claude Code 和 OpenClaw 更像平台级/操作系统级 agent，适合学习 harness、权限、记忆和多任务。
- Codex CLI 更适合学习工程协作质量、前端体验约束和最终交付格式。
- opencode 适合学习高自主工作流，也适合作为“过强约束可能带来什么风险”的讨论样本。
- Kimi CLI 适合学习计划模式、任务列表和工具使用边界。
- Hermes Agent 适合学习工具目录、浏览器能力、委派和记忆卫生。
- Pi 适合作为最小 prompt baseline，用来讲“从 20 行 prompt 到平台级 prompt”的演进。

## PM 迁移问题

- 我的产品需要哪几列是 `强`？
- 哪些模块应该由 prompt 约束，哪些应该由产品机制强制？
- 当前 prompt 最大短板是工具、权限、记忆，还是验证？
- 如果只补一个模块，哪个最能降低用户风险？
